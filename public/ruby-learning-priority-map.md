# Ruby学習の優先度マップ：AI時代に何をどこまで理解すべきか？

## 5段階レベルの定義

| レベル | 定義                                         | AIとの関係               |
| ------ | -------------------------------------------- | ------------------------ |
| **1**  | 知っているだけでいい                         | AIに完全に任せてOK       |
| **2**  | 概念を説明できればいい                       | AIに任せつつ概念は把握   |
| **3**  | コードを読んで理解できる必要がある           | AIの出力を読解できる     |
| **4**  | AIの出力を評価・修正できる必要がある         | AIをレビュー・修正できる |
| **5**  | 絶対に自分で理解し、説明できなければならない | AIを設計・制御できる     |

## 各トピックの詳細

### 1. プログラムの3つの流れ（順次・条件分岐・繰り返し）→ レベル4

**なぜこのレベル？**
プログラミングの基礎中の基礎。AIはこれらの処理を自動生成してくれるが、そのロジックが「ビジネス要件を満たしているか」「バグや無限ループを引き起こさないか」を判断するのは人間。

**チェックポイント：**

- 条件分岐：AIが複雑な要件を誤解して間違ったコードを生成することがよくある
- 繰り返し：無限ループの可能性やパフォーマンスの問題がないかチェック
- 例：「ユーザーがログインしている場合のみ表示」で、ログアウト状態の処理が抜けていないか

---

### 2. データ型（変数・文字列・数値・配列・ハッシュ・真偽値）→ レベル4

**なぜこのレベル？**
データ型はシステム全体のデータモデリングの基礎。データモデリングはAIが苦手とする全体最適に関わるため、人間が担うべき領域。

**チェックポイント：**

- AIが提案したデータ構造が「ビジネス要件を満たしているか」「拡張性があるか」を評価
- 特に配列とハッシュはRailsで頻繁に使われる
- 例：「ユーザー情報をハッシュで管理しているが、配列の方が適切ではないか」

---

### 3. メソッド（引数・戻り値）→ レベル4

**なぜこのレベル？**
メソッド名や引数の順番を丸暗記する必要はない（AIに聞けばすぐ分かる）。しかし「責務の分離」を設計・評価するのは人間の仕事。

**チェックポイント：**

- AIが作ったメソッドが適切な粒度になっているか
- 単一責任の原則を満たしているか
- 例：「このメソッドは、データの取得と加工の両方をしているが、分けた方が良いのではないか」

---

### 4. クラスとオブジェクト（クラス・インスタンス・インスタンス変数）→ レベル5 ⭐最重要

**なぜこのレベル？**
AI時代に最も重要なスキルの一つ。オブジェクト指向やデザインパターンの理解がないと、AIがカオスなコードを量産しシステムが破綻する。

**チェックポイント：**

- オブジェクト指向は「複雑さを管理するための思想」
- SOLID原則（単一責任、オープン・クローズド等）はAIを制御するための「ハーネス」
- RailsのMVCアーキテクチャそのものがオブジェクト指向
- 例：「このクラスの責務は明確か」「ユーザー情報の管理と注文処理は分けるべきか」

---

### 5. 配列操作メソッド（each / map / select / find）→ レベル3

**なぜこのレベル？**
スペルや使い方の完璧な暗記は全く不要。AIが生成したコードを読む際に「チャンク」として瞬時に読んで理解できることが重要。

**覚えるべき役割の違い：**

- `each`: 繰り返し処理（副作用目的）
- `map`: 変換して新しい配列を作る
- `select`: 条件に合うものだけを抽出
- `find`: 条件に合う最初の1つを取得

**判断基準：** AIがeachで書いてきたけど「変換した結果がほしい」なら、「それmapじゃないの？」と言えればOK。

---

### 6. ハッシュ操作・params → レベル4

**なぜこのレベル？**
Railsのparams処理はセキュリティの要。AIにフォーム処理を書かせた際、意図しないデータまで保存してしまう脆弱性がないかを厳密に評価する力が必要。

**チェックポイント：**

- Strong Parametersの設定が甘くないか
- マスアサインメント脆弱性に注意
- 例：「adminフラグまで許可していないか」「ネストしたパラメータの許可設定は正しいか」

---

### 7. エラーの種類と対処 → レベル4

**なぜこのレベル？**
エラーメッセージを見て「おそらく原因はここだろう」と自分で仮説を立てる力が必須。

**主要なエラーと原因：**
| エラー種別 | 意味 | よくある原因 |
|-----------|------|------------|
| SyntaxError | 文法ミス | endの重複、カッコの閉じ忘れ |
| NoMethodError | 存在しないメソッドを呼んだ | nilに対してメソッドを使った |
| ArgumentError | 引数の数が違う | メソッドに渡す値の数が合っていない |

**良い指示 vs 悪い指示：**

- ❌ 「エラーが出ました。どうすればいいですか?」
- ✅ 「NoMethodErrorが出ました。おそらく変数がnilになっているのが原因だと思いますが、確認方法を教えてください」

---

### 8. モジュール / ミックスイン → レベル3

**なぜこのレベル？**
DRY原則や単一責任の原則という設計概念を理解し、AIが生成したコードの構造を読んで把握できれば、具体的な記述方法はAIに任せてOK。

**Railsでの実例：**

- ApplicationControllerにhelper_method :current_userを書いて、全ビューでcurrent_userが使えるようになった仕組みは、Rubyのモジュール/ミックスインの考え方がベース
- RailsではConcernという形でモジュールが頻繁に使われる

---

### 9. AIへの指示パターン → レベル2

**なぜこのレベル？**
「プロンプト集は捨てていい」「プロンプトの呪文を暗記する価値は薄れている」。問題を構造化して伝える概念さえ理解していれば十分。

**意識すべき構造（自然な日本語でOK）：**

1. 何をしたいのか（目的）
2. 現状どうなっているのか
3. 何が問題なのか
4. 自分の仮説は何か

---

### 10. Ruby vs Rails の位置づけ → レベル5 ⭐最重要

**なぜこのレベル？**
Railsの「設定より規約（Rails Way）」はAIを暴走させないための最強の「ハーネス（安全装置）」。

**RubyとRailsの違い：**

- Ruby = プログラミング言語（道具）
- Rails = Webアプリケーションフレームワーク（設計図と道具のセット）

**Rails Wayの具体例：**

- モデル名は単数形、テーブル名は複数形
- コントローラーのアクションは7つの標準アクション（index, show, new, create, edit, update, destroy）
- ファイルの配置場所は決まっている（app/models, app/controllers, app/views）

**AIとの関係：**

- ❌ AIが自由に書いたコード → ファイルの配置がバラバラ、命名規則が統一されていない
- ✅ Rails Wayに沿ったコード → 規約通りの配置、統一された命名、ActiveRecordの規約に従う

## 活用方法

1. **学習の優先順位付け**: 時間が限られているときは、レベル5と4を最優先で理解する
2. **AIとの協働の指針**: レベル5と4は自分で判断しAIを制御する。レベル3以下はAIに任せてOK
3. **復習の基準**: 定期的にレベル5と4の項目を復習し、本質的な理解を深める
